# 嵌入式入门课程学习日志

学习时间：2026年6月1日  
学习主题：嵌入式系统入门与基础实验  
课程模块：E1 `HelloWorld`、E2 数字输出与闪灯、E3 数字输入与按键、E4 模拟输出与呼吸灯、E5 模拟输入与光敏控制、E6 WiFi、E7 舵机控制、E8 WiFi舵机联动  

## 一、学习目标

本次学习的目标主要有以下几个方面：

1. 初步理解什么是嵌入式系统，以及单片机在控制系统中的作用。
2. 学会搭建基础开发环境，完成程序下载、串口监视和结果观察。
3. 掌握 GPIO 的基本使用方法，理解数字输入和数字输出的区别。
4. 学会使用 PWM 完成模拟输出效果，实现 LED 呼吸灯等动态控制。
5. 学会读取传感器的模拟量数据，并根据采样值做出控制决策。
6. 初步接触 WiFi 联网功能，理解嵌入式设备联网的基本流程。
7. 认识舵机控制原理，掌握通过 PWM 信号控制角度的方法。
8. 尝试将联网控制与执行器控制结合起来，形成完整的“小系统”思路。

## 二、课程总体理解

通过本次课程学习，我对“嵌入式”这个概念有了更直观的认识。嵌入式系统并不是单纯写代码，而是“软硬件结合”的开发方式。它通常由控制器、传感器、执行器、供电模块和通信模块组成。程序运行在资源有限的芯片中，因此开发时既要考虑逻辑功能，也要兼顾运行效率、引脚资源、时序要求和稳定性。

和普通上位机程序相比，嵌入式开发更强调：

1. 与硬件打交道，代码执行结果通常对应真实的物理现象。
2. 每个 IO 引脚都有明确用途，配置错误会直接导致实验失败。
3. 调试方式更依赖串口输出、示波、模块状态灯和硬件接线检查。
4. 对时序非常敏感，例如舵机、PWM、串口、WiFi连接都离不开时间控制。

本课程的安排也比较符合入门规律：先从最简单的程序打印和点灯开始，再过渡到按键、模拟量、联网和舵机，最终把多个模块连接成完整应用。这种学习路径让我能从“看见结果”中建立信心，再逐步理解背后的原理。

## 三、分模块学习记录

## 1. E1：HelloWorld

### 学习内容

第一个实验的核心是完成开发板与电脑之间的连接，成功编译程序、下载程序，并在串口监视器中看到 `HelloWorld` 类似的输出内容。这个实验虽然最简单，但意义非常大，因为它验证了整个开发环境是否可用。

### 我的理解

在这个实验中，我重点理解了以下几点：

1. 开发板不是“插上就能跑”，必须先有对应的驱动、开发板配置和串口识别。
2. 程序下载成功，只代表代码进入了芯片，不代表逻辑一定正确。
3. 串口打印是嵌入式入门阶段最重要的调试手段之一，可以帮助开发者确认程序是否进入某一段逻辑。

### 关键知识点

1. `setup()` 一般用于初始化，只执行一次。
2. `loop()` 用于循环执行主逻辑。
3. 串口初始化通常需要配置波特率，例如 `Serial.begin(115200);`
4. 如果串口助手波特率设置错误，即使程序正常，也会看到乱码。

### 学习收获

这个实验让我建立了最基础的开发闭环：写代码、编译、下载、运行、观察输出。只有把这一步打通，后面所有外设实验才有基础。

## 2. E2：数字输出，闪灯

### 学习内容

这一部分主要学习数字输出，也就是让某个引脚输出高电平或低电平，用于控制 LED 灯亮灭。实验现象通常是让 LED 周期性闪烁。

### 我的理解

LED 闪灯实验本质上是在学习“控制输出”，它是后续驱动蜂鸣器、继电器、模块使能脚等外设的基础。通过这个实验，我理解了一个非常重要的概念：程序中的逻辑状态会直接映射成硬件引脚电平，而硬件电平又会转化为物理现象。

### 关键知识点

1. `pinMode(pin, OUTPUT)` 用于把引脚设置为输出模式。
2. `digitalWrite(pin, HIGH)` 表示输出高电平。
3. `digitalWrite(pin, LOW)` 表示输出低电平。
4. `delay()` 可用于实现简单定时，但它会阻塞程序。

### 实验现象分析

当程序让 LED 每隔固定时间切换一次状态时，就会出现闪烁效果。如果闪烁速度过快，人眼会感觉像常亮；如果延时时间较长，则闪烁现象明显。由此我也认识到“时间参数”会直接影响硬件表现。

### 学习收获

我掌握了最基础的 IO 输出控制方法，也明白了后续很多模块其实都是“更复杂的输出控制”。

## 3. E3：数字输入，按键

### 学习内容

这一部分主要学习读取按键状态，根据按键是否按下，执行不同控制逻辑，例如按下按键点亮 LED，松开按键关闭 LED，或者按一次切换一次状态。

### 我的理解

数字输入和数字输出正好相反。输出是“我告诉硬件做什么”，输入是“我读取硬件当前状态”。按键虽然简单，但它体现了嵌入式开发中的“人机交互”思想，也为后面的菜单控制、模式切换、事件触发打下基础。

### 关键知识点

1. `pinMode(pin, INPUT)` 或 `INPUT_PULLUP` 用于配置输入模式。
2. `digitalRead(pin)` 可读取引脚电平状态。
3. 按键常见问题是抖动，因此实际工程中通常需要软件消抖或硬件消抖。
4. 上拉输入模式比较常用，因为能减少引脚悬空导致的误判。

### 问题理解

如果按键直接读取而不做处理，可能会出现按一次触发多次的现象，这就是“抖动”。这说明嵌入式开发不仅要写出能运行的代码，还要考虑信号是否稳定可靠。

### 学习收获

通过按键实验，我对“输入采集”和“事件触发”有了初步理解，也意识到真实硬件信号并不像理论值那样理想。

## 4. E4：模拟输出，呼吸灯

### 学习内容

这一部分主要学习通过 PWM 实现近似模拟输出，让 LED 出现由暗到亮、再由亮到暗的呼吸效果。

### 我的理解

单片机大部分引脚本质上只能输出高低电平，但通过快速切换高低电平并改变占空比，就可以在视觉上形成“亮度变化”的效果。这让我第一次接触到“数字手段实现模拟效果”的思想。

### 关键知识点

1. PWM 不是直接输出连续变化电压，而是通过占空比变化来模拟平均效果。
2. 占空比越大，LED 看起来越亮。
3. PWM 不只可以控制 LED 亮度，也常用于电机调速、蜂鸣器发声、舵机控制等。
4. 呼吸灯实验能很好地帮助理解“渐变控制”和“循环逻辑”。

### 学习收获

我进一步认识到，嵌入式开发中的很多“模拟控制”其实都建立在定时器和 PWM 基础之上，这一块是后续学习的重要基础。

## 5. E5：模拟输入，光敏电阻感受光

### 学习内容

这一部分主要是读取光敏电阻的模拟值，根据环境光变化得到不同采样结果，并可进一步实现“自动亮灯”或“亮度联动控制”等功能。

### 我的理解

这个实验让我从“控制输出”迈向“感知环境”。如果说 LED、舵机这些模块是执行器，那么光敏电阻就是传感器。嵌入式系统要真正“智能”，就必须先获取环境信息，再根据程序逻辑做出响应。

### 关键知识点

1. 模拟输入通常通过 ADC 完成。
2. 传感器输出不是简单的高低电平，而是一段连续变化的数值范围。
3. 读取到的数据往往需要做阈值判断、滤波或者映射处理。
4. 同样的程序，在不同光照环境下，读数会发生明显变化。

### 延伸理解

如果把光敏电阻替换成温度传感器、气体传感器、土壤湿度传感器，程序思路其实是类似的：采样、判断、控制。因此这个实验虽然简单，但具有很强的代表性。

### 学习收获

我明白了传感器并不是“直接输出答案”，而是输出一个原始信号，开发者需要根据这个信号设计判断逻辑。

## 6. E6：WiFi

### 学习内容

WiFi 模块实验主要是让开发板接入无线网络，完成联网初始化，并为后续的数据传输、远程控制打基础。

### 我的理解

这一部分让我意识到，现代嵌入式开发已经不仅是本地控制，还越来越强调联网能力。设备连上 WiFi 后，就能与手机、电脑、云端服务进行通信，从而扩展出更多应用场景，例如远程监控、远程开关、数据上传、物联网控制等。

### 关键知识点

1. WiFi 连接需要 SSID 和密码。
2. 程序中通常需要轮询连接状态，直到连接成功。
3. 联网实验要特别注意串口打印，因为连接失败、密码错误、信号弱都需要通过日志定位。
4. 网络功能往往受环境影响较大，代码正确不代表一定马上连接成功。

### 学习收获

我认识到，嵌入式系统一旦具备联网能力，开发思维就会从“单机控制”升级为“系统协同”，这也是物联网技术的基础。

## 7. E7：控制舵机

### 学习内容

这一部分主要学习舵机的控制方法，了解舵机根据控制脉冲调整角度的工作方式，实现指定角度旋转、往返摆动等动作。

### 我的理解

舵机实验的意义在于，它让抽象的代码变成了更加明显的机械动作。LED 只是亮或灭，而舵机是“真实运动”的执行器，更接近机器人、自动门、机械臂等实际应用。

### 关键知识点

1. 舵机控制依赖 PWM 或定时脉冲信号。
2. 控制角度时要注意范围限制，避免超出机械行程。
3. 舵机动作时可能出现抖动，这既可能与供电有关，也可能与控制信号不稳定有关。
4. 电源设计在执行器控制中很重要，不能只关注代码逻辑。

### 学习收获

这个实验让我真正感受到嵌入式控制“让机械动起来”的魅力，也意识到硬件供电、电流和稳定性在工程中同样重要。

## 8. E8：WiFi舵机

### 学习内容

该部分是一个综合实验，把前面学过的 WiFi 与舵机结合起来，实现远程控制舵机动作。它体现了“通信 + 控制”的完整应用思路。

### 我的理解

这是整套入门课程中很有代表性的综合案例。前面的实验分别解决了感知、输出、通信、执行中的单个环节，而这里开始把这些能力串联起来。通过这个实验，我对嵌入式系统的整体架构有了更清晰的认识：

1. 先连接网络，建立通信通道。
2. 接收控制命令或控制数据。
3. 对命令进行解析。
4. 驱动舵机执行动作。
5. 必要时再把状态反馈出去。

### 学习收获

我感觉自己不再只是做单一实验，而是在尝试搭建一个小型完整系统。这种系统化思维对后续学习非常重要。

## 四、学习中理解到的核心知识

通过本次课程，我总结出嵌入式入门阶段最重要的几个核心知识：

1. `GPIO` 是最基础的控制入口，几乎所有外设都绕不开它。
2. `输入` 与 `输出` 是最基本的逻辑模型，所有复杂应用都可以看成这两者的扩展。
3. `PWM` 是连接数字控制与模拟效果的重要桥梁。
4. `ADC` 是获取环境信息的重要手段，是传感器应用基础。
5. `串口调试` 是入门阶段最有效的问题定位方式。
6. `联网能力` 决定了设备是否能走向物联网应用。
7. `执行器控制` 让程序真正对物理世界产生影响。
8. `软硬件协同` 才是嵌入式开发真正的核心，不是单纯会写代码就够了。

## 五、实验代码理解记录

为了巩固知识，我对实验程序的结构也进行了总结。大多数基础实验的代码都遵循类似框架：

```cpp
void setup() {
  // 初始化串口
  // 初始化引脚模式
  // 初始化外设
}

void loop() {
  // 读取输入
  // 根据条件判断
  // 控制输出
  // 适当打印调试信息
}
```

这个结构看似简单，但已经包含了嵌入式开发的基本思想：

1. 先完成初始化。
2. 在循环中不断采集状态。
3. 根据状态执行控制逻辑。
4. 持续运行，形成闭环。

## 六、常见问题与排查记录

在学习这类课程时，我认为以下问题最容易出现，也最值得记录：

### 1. 找不到串口

可能原因：

1. 数据线只支持充电，不支持数据传输。
2. 驱动未安装或安装不正确。
3. 开发板没有正常上电。
4. 电脑端口被占用。
5. 板卡型号或串口选择错误。

排查思路：

1. 更换数据线和 USB 接口。
2. 查看设备管理器是否识别到串口设备。
3. 检查开发环境中的板卡和端口设置。
4. 按复位键重新尝试下载。
5. 通过最简单的串口打印程序验证环境。

### 2. 程序下载成功但无现象

可能原因：

1. 引脚定义错误。
2. 硬件接线错误。
3. 供电不足。
4. 程序逻辑没有进入目标分支。
5. 外设损坏或接触不良。

排查思路：

1. 先看串口输出是否正常。
2. 再核对引脚号和接线图。
3. 最后检查模块本身是否工作正常。

### 3. 按键触发不稳定

原因通常与按键抖动、引脚悬空或逻辑设计不合理有关。解决方法包括：

1. 使用上拉输入模式。
2. 加入软件消抖延时。
3. 使用状态机方式处理按键事件。

### 4. 舵机抖动或不转

可能原因：

1. PWM 信号不稳定。
2. 供电电流不足。
3. 地线未共地。
4. 控制角度超出范围。

这个问题让我认识到，执行器类模块对硬件条件要求更高，不能只从代码角度思考。

通过这次嵌入式入门课程学习，我最大的感受是：嵌入式开发非常强调“实践驱动理解”。很多概念如果只看文字会觉得抽象，但一旦结合 LED、按键、光敏电阻、WiFi、舵机这些实验，理解速度会明显提升。

不过通过这次课程，我已经建立起了比较清晰的入门认知：

1. 嵌入式不是只会写程序，而是写“能驱动硬件”的程序。

3. 调试能力和排查思路与代码能力同样重要。
